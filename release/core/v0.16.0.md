# @synkro/core v0.16.0

**Release date:** 2026-03-09

## Changes

### IMP-05: Structured logging

Logger now supports a `logFormat` option (`"text"` or `"json"`). In JSON mode, each log entry is a single-line JSON object with `level`, `msg`, `timestamp`, and contextual fields like `requestId`, `eventType`, and `workflowName`.

```typescript
const synkro = await Synkro.start({
  debug: true,
  logFormat: "json",
});
// Output: {"level":"debug","msg":"[HandlerRegistry] handleMessage","eventType":"user:created","requestId":"req-123","entries":1,"timestamp":"..."}
```

All internal log call sites updated to pass structured fields. Text mode (default) remains backward-compatible.

### FT-09: Event filtering / conditional handlers

Handlers can now specify a `filter` predicate that runs before execution. If the filter returns `false`, the handler is skipped — no failure event, no retry, no metrics impact.

```typescript
const synkro = await Synkro.start({
  events: [
    {
      type: "order:created",
      handler: handleHighValueOrder,
      filter: (payload) => payload.total > 1000,
    },
  ],
});

// Or dynamically:
synkro.on("order:created", handler, undefined, undefined, (payload) => payload.total > 1000);
```

When all handlers for an event are filtered out, the message is silently dropped (no completion/failure events, no metrics tracked).

### FT-01: Dead letter queue for failed events

Failed events (after retry exhaustion) can now be persisted to a dead letter queue for later inspection and replay. Opt-in via configuration:

```typescript
const synkro = await Synkro.start({
  deadLetterQueue: true,
});

// Query failed events
const items = await synkro.getDeadLetterItems("user:created", { limit: 10 });

// Replay a failed event
await synkro.replayDeadLetterItem(items[0]);

// Clear the DLQ for an event type
await synkro.clearDeadLetterQueue("user:created");
```

Each `DeadLetterItem` includes `eventType`, `requestId`, `payload`, `errors`, `failedAt` (ISO timestamp), and `attempts` count. DLQ items persist until explicitly cleared. Workflow step failures are excluded from the DLQ (they are handled by the workflow engine).

### Transport interface changes

`TransportManager` extended with three new methods: `pushToList()`, `getListRange()`, and `deleteKey()`. Both `RedisManager` and `InMemoryManager` implement them. Custom transport implementations will need to add these methods.
